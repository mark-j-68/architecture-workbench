# North Star Demo 0.2

Release 0.2 demonstrates Discovery Foundations: deterministic evidence
collection and structural observation for an existing Spring Boot repository.

The demo length is 15 minutes.

## Demo Message

"Architecture Workbench Release 0.2 observes a repository. It collects evidence,
records provenance, calculates confidence, and publishes deterministic
observations into the Architecture Intelligence Model. The intelligence layer
then creates findings, recommendations, and proposed changes. Discovery itself
does not mutate the graph."

## Flow

### 0:00-1:00 - Create Workspace

The user creates a workspace for an existing Spring Boot system.

Expected state:

- workspace is created
- graph is initially empty or minimal
- discovery action is available

### 1:00-2:00 - Register Repository

The user registers a local Spring Boot repository path.

Expected state:

- source is registered
- source validation confirms repository path is readable
- discovery run has correlation metadata

### 2:00-4:00 - Run Discovery

The user starts discovery.

Expected state:

- repository plugin inventories files and directories
- Maven plugin detects modules
- Java plugin detects packages, classes, interfaces, annotations
- Spring plugin detects controllers, services, repositories, configuration
- dependency plugin detects package dependencies
- contract plugin detects OpenAPI or controller-derived API endpoints
- messaging plugin detects producer/consumer/topic evidence where present

### 4:00-5:30 - Show Maven Modules

Expected UI state:

- detected root `pom.xml`
- detected child modules
- module coordinates and dependencies
- evidence links to `pom.xml` files

Message:

"This is observed build evidence. It is not yet a judgement about architecture
quality."

### 5:30-7:00 - Show Spring Components

Expected UI state:

- controllers
- services
- repository components
- configuration components
- source file provenance
- annotation evidence

Message:

"The framework plugin can make deterministic observations from annotations. It
does not decide whether the architecture is good."

### 7:00-8:30 - Show Package Dependencies

Expected UI state:

- package dependency graph
- imports or dependency declarations as evidence
- confidence values for each relationship

Message:

"Package dependencies become evidence for later coupling analysis. Product-level
interpretation is a Release 0.3 concern."

### 8:30-10:00 - Show API Contracts

Expected UI state:

- OpenAPI contracts if present
- controller-derived endpoint evidence
- unmatched controller/spec observations if supported
- provenance and confidence

Message:

"Contracts are detected as evidence. Contract maturity is interpreted later by
the intelligence layer."

### 10:00-11:00 - Show Messaging Relationships

Expected UI state:

- producer evidence
- consumer evidence
- queue/topic/event bus references
- event or command contract evidence where present

Message:

"Messaging discovery observes producers, consumers, and topics. It does not yet
claim that the product is event-driven or a distributed monolith."

### 11:00-12:00 - Inspect Evidence Provenance And Confidence

Expected UI state:

- evidence source path
- plugin id and version
- checksum/reference where available
- observed/deterministic/heuristic classification
- confidence value and rationale

Message:

"Confidence exposes uncertainty. It must not hide weak evidence behind a
precise-looking score."

### 12:00-13:30 - Generate Findings Through AIM

The existing intelligence workflow consumes evidence and observations.

Expected state:

- AIM Evidence exists
- AIM Observations exist
- AIM Findings are generated by intelligence services
- findings retain evidence traceability

Message:

"Findings are not scanner output. They belong to the Architecture Intelligence
Model."

### 13:30-14:30 - Show Proposed Changes

Expected state:

- proposed changes are visible
- each proposed change traces to recommendation, finding, observation, and
  evidence
- graph has not changed automatically

Message:

"Discovery suggests nothing directly to the graph. Inferred graph changes go
through the proposed change workflow."

### 14:30-15:00 - Lead Into Release 0.3

Closing message:

"Release 0.2 gives us trustworthy evidence from one repository. Release 0.3
will compose many of these repository discoveries into Product Architecture
Intelligence: modularity, release independence, contract maturity, and
distributed monolith risk."

## Success Criteria

The demo succeeds when the audience sees:

- deterministic repository discovery
- evidence provenance
- confidence calculation
- structural observations
- AIM traceability
- findings and recommendations produced outside scanner plugins
- proposed changes that do not mutate the graph automatically
- a clear path from single-repo evidence to multi-repo product reasoning
