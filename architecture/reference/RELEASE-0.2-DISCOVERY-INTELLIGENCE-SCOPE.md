# Release 0.2 Discovery Foundations Scope

Release 0.2 is the first capability-focused release after the Architecture OS
kernel foundation. Its purpose is to make Discovery Mode useful for existing
Java/Spring Boot systems by collecting evidence and deterministic structural
observations.

Terminology note: earlier planning used "Discovery Intelligence" for this
release. The refined boundary is:

- v0.2: Discovery Foundations / Evidence Collection
- v0.3: Multi-Repo Product Architecture Intelligence

## Product Goal

An architect can point Architecture Workbench at an existing Spring Boot
repository and receive evidence-backed structural observations. The Architecture
Intelligence Model then interprets those observations into findings,
recommendations, and proposed changes.

## In Scope

### Java/Spring Boot Project Analysis

Release 0.2 focuses on local Java/Spring Boot repositories. Analysis is static
and local. It does not require cloud access, GitHub access, runtime telemetry,
or AI provider calls.

### Maven Module Discovery

Discovery identifies:

- root `pom.xml`
- child Maven modules
- module names and paths
- multi-module structure
- likely application, domain, adapter, and infrastructure modules

### Package Dependency Detection

Discovery should detect package-level relationships well enough to identify:

- dependencies between packages
- likely layer dependencies
- dependency direction concerns
- potential cycles
- highly coupled packages

Deep semantic static analysis is not required for Release 0.2, but the
dependency model should be good enough to support useful architecture findings.

### Spring Component Detection

Discovery detects Spring stereotypes and API entry points, including:

- controllers
- services
- repositories
- configuration classes
- application bootstrap classes

### Inferred Layers

Discovery infers likely layers from packages, annotations, and file paths:

- API/controller layer
- application/service layer
- domain layer
- persistence/repository layer
- configuration/infrastructure layer
- test layer

### Inferred Architecture Style Signals

Discovery collects signals that the intelligence layer can use to infer broad
architecture style, such as:

- layered monolith
- modular monolith
- hexagonal or ports-and-adapters candidate
- service-oriented or API-first structure
- unclear or mixed style

The Discovery Engine should not make definitive architecture style
classifications.

### Basic Bounded Context Candidates

Discovery suggests bounded context candidates from:

- Maven modules
- package roots
- domain terms
- controller/resource groupings
- aggregate-like class names
- service/repository clusters

Candidates must remain proposed intelligence, not automatic graph mutation.

### DDD Smell Signals

Release 0.2 should collect evidence that can support basic DDD and architecture
smell findings, including:

- missing domain layer
- anemic package structure
- controller-to-repository coupling
- service classes acting as transaction scripts
- unclear bounded context boundaries
- missing ubiquitous language evidence
- missing ADRs for significant architectural decisions

### Coupling And Cycle Detection

Discovery should identify:

- package cycles
- module cycles where detectable
- excessive inward dependencies on infrastructure
- controller or API dependencies on persistence concerns
- high fan-in or fan-out packages

### Architecture Score Inputs

Release 0.2 collects score inputs for demo and prioritization. The score itself
belongs to the intelligence layer.

The score should be evidence-backed and explainable. It should reflect signals
such as:

- test presence
- README/docs presence
- ADR presence
- layer clarity
- coupling and cycles
- Spring component organization
- bounded context clarity

The score is a decision aid, not a compliance grade.

### Evidence-Backed Recommendation Inputs

Recommendations are generated by the intelligence layer and must be traceable:

Evidence -> Observation -> Finding -> Recommendation -> Proposed Change.

Every recommendation should include:

- rationale
- confidence
- related findings
- supporting evidence
- estimated impact
- estimated effort
- proposed next action

## Explicitly Out Of Scope

Release 0.2 does not include:

- live AI providers
- database persistence
- cloud runtime scanning
- GitHub PR integration
- meeting capture
- event sourcing
- production authentication or authorization
- multi-user collaboration
- provider marketplace
- automatic graph mutation from discovery
- automatic architecture decisions

## Success Criteria

Release 0.2 is successful when the North Star demo can show:

- a Spring Boot project discovered from local files
- evidence, observations, findings, recommendations, and proposed changes
- a Review Board workflow over a recommendation and proposed change
- explicit acceptance mutating the graph
- updated projection after acceptance
- traceability from graph change back to evidence
- audit records and persistence integrity for local/demo use

## Design Constraints

Discovery Foundations must preserve the platform constitution:

- graph remains canonical
- discovery outputs are intelligence objects and proposed changes
- all graph mutation goes through validated application services
- Review Board recommendation does not automatically mutate the graph
- local JSON persistence remains an adapter
- event sourcing remains deferred
